# System Requirements

This application is a Laravel 7.30.4 project currently running on PHP 7.4.33. Keep the production server close to these versions unless you are planning a framework upgrade.

## Server

- Ubuntu 24.04.4 LTS, Noble
- Apache 2.4 or Nginx, with the web root pointed to `public/`
- PHP 7.4.33 for CLI and the web server runtime
- MySQL or MariaDB for the application database
- RabbitMQ for the M-Pesa C2B queue worker
- `systemd` or another process manager for long-running queue commands

## New Installations

For a fresh Ubuntu 24.04 server, use the stack installer at `scripts/install-stack.sh`. It installs Apache, MySQL, Redis, RabbitMQ, PHP 7.4 from the Ondrej PPA, required PHP extensions, Apache rewrite support, Redis ACLs, and a RabbitMQ admin user. If the app is already cloned, it also creates or syncs the Laravel `.env` file from `.env.example`, including `APP_DIR`.

After the project is present on the server, use `scripts/setup-laravel-apache.sh` to configure the Apache vhost, Laravel `.env`, MySQL database/user, Composer dependencies, permissions, migrations, and optional HTTPS.

The installer deliberately does not contain a real password. Pass the password at runtime:

```bash
sudo APP_DIR=/var/www/m.pbsys.co.ke APP_USER=timon APP_PASS='set-a-strong-password' bash scripts/install-stack.sh
```

Or copy it to `/opt` first:

```bash
sudo install -m 700 scripts/install-stack.sh /opt/install-stack.sh
sudo APP_DIR=/var/www/m.pbsys.co.ke APP_USER=timon APP_PASS='set-a-strong-password' bash /opt/install-stack.sh
```

Apache and Laravel setup example:

```bash
sudo DOMAIN=m.pbsys.co.ke \
  PROJECT_DIR=/var/www/m.pbsys.co.ke \
  APP_DIR=/var/www/m.pbsys.co.ke \
  DB_NAME=mlp_pbsys \
  DB_USER=mlp_pbsys \
  DB_PASS='set-a-strong-password' \
  SSL_EMAIL=admin@example.com \
  ENABLE_SSL=1 \
  bash scripts/setup-laravel-apache.sh
```

Useful runtime options:

- `APP_DIR=/var/www/m.pbsys.co.ke`
- `APP_URL=https://m.pbsys.co.ke`
- `DB_DATABASE=your_database`
- `DB_USERNAME=your_database_user`
- `DB_PASSWORD=your_database_password`
- `REDIS_PASSWORD=your_redis_password`
- `RABBITMQ_USER=your_rabbitmq_user`
- `RABBITMQ_PASSWORD=your_rabbitmq_password`
- `MPESA_TRANSACTION_STATUS_INITIATOR=your_initiator`
- `MPESA_TRANSACTION_STATUS_CREDENTIAL=your_encrypted_credential`
- `MPESA_TRANSACTION_STATUS_IDENTIFIER=shortcode`
- `SSL_EMAIL=admin@example.com`
- `ENABLE_SSL=1`

Use the same password values in the server `.env` for `DB_PASSWORD`, `REDIS_PASSWORD`, and `RABBITMQ_PASSWORD` only if that is the intended deployment policy. For stronger separation, create different passwords per service.

Do not commit real installation passwords to GitHub.

## PHP Extensions

Install these PHP 7.4 extensions before running Composer:

- `bcmath`
- `ctype`
- `curl`
- `dom`
- `fileinfo`
- `gd`
- `json`
- `mbstring`
- `openssl`
- `pdo_mysql`
- `pdo_sqlite`, needed for CI and local tests
- `tokenizer`
- `xml`
- `zip`

On Ubuntu, package names usually look like `php7.4-curl`, `php7.4-mbstring`, and so on.

## Composer

This project keeps `composer.phar` in the repository so the server can install dependencies even when Composer is not installed globally.

Use:

```bash
php composer.phar install --no-dev --optimize-autoloader
```

For development or CI-style installs, use:

```bash
php composer.phar install
```

Do not commit `vendor/`. It is generated by Composer on the server.

## Frontend Build

The project uses Laravel Mix 5 and an older Webpack toolchain. Use Node.js 14 with npm 6 when building assets:

```bash
npm ci
npm run prod
```

Do not commit `node_modules/`.

## Environment File

Create the server `.env` from the safe template:

```bash
cp .env.example .env
php artisan key:generate
```

Then fill in the real values directly on the server. Required groups include:

- `APP_*` application settings
- `DB_*` database settings
- `RABBITMQ_*` queue settings
- `MPESA_TRANSACTION_STATUS_*` transaction status settings
- `MAIL_*` mail settings
- Any `AWS_*` or `PUSHER_*` values if those services are enabled

Never commit `.env` or machine-specific files from `.vscode/`.

`APP_DIR` should match the absolute path where the repository lives on the server. The installer uses it to find `.env.example`, create `.env`, and write the same `APP_DIR` value back into `.env`.

## Laravel Runtime

After dependencies and `.env` are ready:

```bash
php artisan config:clear
php artisan cache:clear
php artisan route:clear
php artisan migrate --force
```

The C2B worker command is:

```bash
php artisan mpesa:process-c2b-queue
```

For a one-message smoke test:

```bash
php artisan mpesa:process-c2b-queue --once
```

Production should run the worker under `systemd`, Supervisor, or another service manager so it restarts after failures and server reboots.
